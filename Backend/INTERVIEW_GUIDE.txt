                           INTERVIEW STUDY GUIDE
                     E-Commerce Backend REST API Project
================================================================================

This guide serves as a comprehensive tool to help you confidently explain the 
architecture, design decisions, and implementation details of this project during 
a Backend/Node.js Software Engineer interview.

--------------------------------------------------------------------------------
1. PROJECT OVERVIEW
--------------------------------------------------------------------------------
This project is a RESTful E-Commerce Backend API engineered using Node.js, 
Express.js, and MongoDB with Mongoose ODM. It implements:
- Core User Authentication (Register/Login via secure JWT-in-cookie flow).
- Product Catalog management with complete CRUD capabilities.
- A fully relational Shopping Cart system with automated price calculations.
- A secure Order Processing module implementing ACID-like transactional checks
  (e.g., verifying catalog stock availability, decrementing inventory, placing
  orders, and resetting the cart state).

--------------------------------------------------------------------------------
2. WHY I BUILT THIS PROJECT
--------------------------------------------------------------------------------
I built this project to demonstrate hands-on competency in:
- Backend Systems Design: Architecting modular Express code with separate routes,
  controllers, models, and middleware.
- Data Modeling: Designing clean schema associations in a NoSQL database (MongoDB)
  using reference-based document relationships.
- Business Logic Orchestration: Building robust transactional workflows (Order placement)
  where database consistency and stock integrity are prioritized.
- Security Best Practices: Storing hashed passwords, generating signable JWT tokens,
  and using cookies to minimize Cross-Site Scripting (XSS) and Cross-Site Request
  Forgery (CSRF) vulnerabilities.

--------------------------------------------------------------------------------
3. DATABASE DESIGN
--------------------------------------------------------------------------------
The database consists of 4 interrelated collections modeled via Mongoose:

a) Users ("user" model):
   - Fields: username (String), email (String, unique), password (String, hashed),
     role (String, enum: ['customer', 'admin']).
   - Purpose: Identity authentication and role-based tracking.

b) Products ("productschema" model):
   - Fields: name (String), description (String), price (Number), stock (Number),
     category (String), image (String).
   - Purpose: The master product catalog containing pricing and available inventory.

c) Carts ("cart" model):
   - Fields: user (ObjectId reference to "user"), items (Array of product-quantity pairs),
     totalPrice (Number).
   - Items schema contains product (ObjectId reference to "productschema") and quantity.
   - Purpose: Stores transient shopper selections. Automatically computes total cost.

d) Orders ("order" model):
   - Fields: user (ObjectId reference to "user"), items (Array of product-quantity pairs),
     totalPrice (Number), status (String, enum: ["Pending", "Delivered", "Cancelled"]).
   - Purpose: Permanent, immutable record of finalized transactions.

5. AUTHENTICATION FLOW
--------------------------------------------------------------------------------
1. Registration: The user sends a POST request with username, email, and password.
   We hash the password using `bcrypt` and save the user. A JWT token is generated 
   and set in the HTTP response cookie named "token".
2. Login: The user posts email/username and password. We find the user, compare the 
   received password with the database hash using `bcrypt.compare()`. If matches, we 
   generate a JWT containing the user id and role.
3. Token Storage: The JWT is attached to the client browser via an HTTP-only 
   cookie (`res.cookie('token', token, { httpOnly: true })`) to block malicious 
   client-side JavaScript access.
4. Route Protection: Protected endpoints query the authentication middleware, 
   verifying if the cookie/bearer token matches our secret key before routing requests.

--------------------------------------------------------------------------------
6. PRODUCT CRUD EXPLANATION
--------------------------------------------------------------------------------
Managed through `src/controllers/product.Controller.js` and `product.router.js`:
- Create (`POST /user/product`): Sanitizes fields, checks if a product name already 
  exists, creates a new product document, and returns 200.
- Read (`GET /user/product`): Performs `productModel.find()` to fetch all active listings.
- Update (`PATCH /user/product/:id`): Finds catalog product via params ID, updates details
  via `findByIdAndUpdate()`, and returns the modified product.
- Delete (`DELETE /user/product/:id`): Searches product via ID and deletes using 
  `findByIdAndDelete()`.

--------------------------------------------------------------------------------
7. CART WORKING
--------------------------------------------------------------------------------
- Add to Cart (`POST /cart`): Accepts `productId` and `quantity`. First, validates 
  the product exists and matches available stock. Then queries if the user already 
  has a cart. If yes, it checks if the product is already in the items array; if so,
  adds the quantity. If not, it pushes the item to the items array.
- Auto-calculation: Loops over all active items, queries their prices from the 
  product catalog database directly (to prevent front-end tampering), sums up 
  `(price * quantity)`, updates `totalPrice`, and saves the document.
- Retrieve (`GET /cart/:userId`): Uses Mongoose `.populate("items.product")` to expand 
  the product ObjectIds into complete product details.
- Remove (`DELETE /cart/:productId`): Filters the items array to exclude the specified 
  product, recalculates `totalPrice` from the remaining items, and saves.

--------------------------------------------------------------------------------
8. ORDER WORKING
--------------------------------------------------------------------------------
Triggered via `POST /order` (Checkout workflow):
1. Cart Retrieval: Reads the cart of the authenticated user and populates product details.
2. Inventory Check: Iterates through each item in the cart, checking if its requested 
   quantity is <= the current stock of that product in the master product catalog.
3. Transactional Simulation:
   - If stock is insufficient, terminates early and returns `400 Bad Request`.
   - If stock is sufficient, decrements the product's catalog stock dynamically in 
     the database: `productModel.findByIdAndUpdate(productId, { $inc: { stock: -quantity } })`.
4. Order Creation: Saves an Order record containing user ID, items (copied from cart), 
   total price, and status "Pending".
5. Cart Reset: Empties the items array of the user's cart, sets `totalPrice` to 0, 
   and saves.

--------------------------------------------------------------------------------
9. JWT AUTHENTICATION EXPLANATION
--------------------------------------------------------------------------------
JSON Web Token (JWT) is a compact, URL-safe standard (RFC 7519) representing claims 
transferred between parties. In our application:
- Payload: Contains `id` (User ObjectId) and `role`.
- Signature: Generated using the payload and a backend-stored `JWT_SECRET`.
- Verification: The `authMiddleware` checks if a token exists in cookies or headers, 
  verifies the signature using `jwt.verify()`, and populates `req.user`. This prevents 
  the client from tampering with their session identity.

--------------------------------------------------------------------------------
10. MONGODB RELATIONSHIPS
--------------------------------------------------------------------------------
We use "Referenced Relationships" (Normalization) to link schemas:
- User - Cart/Order: 1-to-1 (User to Cart) and 1-to-Many (User to Orders) using Mongoose 
  `ObjectId` with `ref: "user"`.
- Product - Cart/Order Items: References the master product collection with `ref: "productschema"`.
This approach avoids duplicated data and ensures updates to product pricing/descriptions 
are immediately reflected when populating active user carts.

--------------------------------------------------------------------------------
11. MONGOOSE METHODS USED
--------------------------------------------------------------------------------
- `findOne(query)`: Retrieves a single document matching criteria (e.g. email/username).
- `create(data)`: Creates and saves a new document in MongoDB.
- `find(query)`: Finds all matching documents (e.g., retrieving user orders).
- `findById(id)`: Fetches a single document by its `_id`.
- `findByIdAndUpdate(id, update, options)`: Modifies a document by ID and returns 
  the updated document (`new: true`).
- `findByIdAndDelete(id)`: Removes a document from the database.
- `populate(path)`: Dynamically replaces referenced ObjectIds with the actual documents.
- `save()`: Persists changes made to model instances back to MongoDB.

--------------------------------------------------------------------------------
12. REST APIs USED
--------------------------------------------------------------------------------
- `POST`: Used for creating resources where idempotency is not guaranteed (e.g., 
  `/auth/user/register`, `/cart` add, `/order` creation).
- `GET`: Safe, idempotent retrieval of resource data (`/user/product`, `/cart/:userId`, 
  `/orders/:userId`).
- `PATCH`: Modifies parts of an existing resource (`/user/product/:id`).
- `DELETE`: Permanently removes resources (`/user/product/:id`, `/cart/:productId`).

--------------------------------------------------------------------------------
13. CHALLENGES FACED
--------------------------------------------------------------------------------
During an interview, describe these technical challenges you successfully resolved:
- Race Conditions & Stock Validation: Ensuring stock isn't overallocated. Solved by 
  performing database validations before writing an order, and using MongoDB atomic 
  updates (`$inc` operator) to decrement stock.
- Automated Cart Totals: Ensuring clients can't forge total prices. Solved by fetching 
  prices directly from the database during recalculation, completely ignoring client-sent
  prices.
- JWT Cookie-Parsing Setup: Extracting tokens seamlessly across headers and cookies. 
  Solved by configuring `cookie-parser` globally and fallback verification logic.

--------------------------------------------------------------------------------
15. POSSIBLE INTERVIEW QUESTIONS & ANSWERS
--------------------------------------------------------------------------------
Q1: Why did you use cookies instead of local storage to store JWTs?
A1: Storing JWT tokens in local storage makes them accessible to client-side scripts,
    leaving them vulnerable to Cross-Site Scripting (XSS) attacks. By storing tokens 
    in HTTP-only cookies, browser scripts cannot read the token, which significantly 
    improves the security of user sessions.

Q2: What happens to the cart checkout if Mongoose fails midway?
A2: Currently, it executes step-by-step. To make this fully production-safe, we can 
    implement Mongoose Transactions (`session`). Transactions guarantee that either 
    all updates (deducting stock, creating the order, clearing the cart) succeed, 
    or they all fail and roll back, preserving database integrity.

Q3: Why did you choose MongoDB over SQL for this project?
A3: E-commerce catalogs often require flexible metadata schemas (different product 
    categories might have varying specifications like size, color, RAM, warranty). 
    MongoDB's document model allows flexible schemas while offering high-performance 
    querying and simple scaling.
